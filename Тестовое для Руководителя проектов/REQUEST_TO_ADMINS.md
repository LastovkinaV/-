# Запрос системным администраторам

Ниже готовый текст, который можно отправить системным администраторам.

## Что нужно заменить перед отправкой

Замените следующие значения:

- `<SUBDOMAIN>` — поддомен для размещения, например `pm-test.company.ru`
- `<OWNER_NAME>` — ваше имя
- `<OWNER_EMAIL>` — ваш email
- `<DEADLINE>` — желаемый срок публикации
- `<ACCESS_MODE>` — `public`, `basic auth`, `ip allowlist` или `vpn only`

## Готовый текст запроса

Тема:

`Разместить статический лендинг на домене <SUBDOMAIN> без индексации`

Текст:

Коллеги, привет.

Нужно разместить статический лендинг на нашем домене:

- домен: `<SUBDOMAIN>`
- владелец: `<OWNER_NAME>`
- контакт: `<OWNER_EMAIL>`
- желаемый срок: `<DEADLINE>`

Это обычный статический сайт без бэкенда и без сборки.

## Что нужно сделать

1. Разместить содержимое папки `Тестовое для Руководителя проектов` как статический сайт.
2. Подключить домен `<SUBDOMAIN>` и настроить HTTPS.
3. Сделать так, чтобы сайт не индексировался поисковыми системами.
4. Дать мне доступ на обновление файлов сайта без участия администраторов.
5. Вернуть мне публичную ссылку и подтвердить, что антииндексация включена.

## Технические параметры сайта

- стек: `Static HTML/CSS/JavaScript`
- серверная логика: отсутствует
- сборка: не нужна
- entry point: `index.html`
- ассеты: `styles.css`, `app.js`, `assets/*`
- дополнительные материалы: `input/messages.md`

## Что важно для антииндексации

В проекте уже добавлено:

- `robots.txt` с `Disallow: /`
- meta-теги:
  - `noindex`
  - `nofollow`
  - `noarchive`
  - `nosnippet`
  - `noimageindex`

Прошу дополнительно выставить на стороне сервера HTTP-заголовок:

`X-Robots-Tag: noindex, nofollow, noarchive, nosnippet, noimageindex`

Это нужно как дополнительная защита от индексации.

## Важная оговорка

Если сайт не просто не должен индексироваться, а вообще не должен быть доступен посторонним, то `robots.txt` и meta-теги недостаточны.

В таком случае прошу сразу выбрать режим доступа:

- `<ACCESS_MODE>`

Рекомендуемый вариант для внутреннего тестового:

- `basic auth`

или

- `ip allowlist`

или

- размещение только во внутреннем контуре / через VPN

## Доступ для самостоятельного обновления

Прошу также выдать мне доступ, чтобы я могла самостоятельно обновлять файлы сайта после публикации.

Подойдут любые из вариантов:

- `SFTP`
- `SSH` + доступ к каталогу сайта
- доступ через файловый менеджер в панели хостинга
- доступ к bucket/object storage, если сайт будет размещен там

Предпочтительный вариант:

- `SFTP` или доступ через панель

Что именно мне нужно:

- адрес подключения
- логин
- способ авторизации
- если нужен пароль, то временный пароль
- если нужен SSH key, инструкция куда прислать публичный ключ
- путь к каталогу сайта на сервере
- информация, как именно публиковать обновления

Если это возможно, прошу дать мне права только на каталог этого сайта, без лишнего доступа к другим проектам.

## Короткая техническая справка для админов

Это сайт без сборки, поэтому его можно размещать:

- через nginx как каталог со статикой
- через apache как каталог со статикой
- через object storage/static hosting
- через любой корпоративный reverse proxy, который умеет отдавать статические файлы

Никаких зависимостей, `npm install`, Docker и runtime не требуется.

## Пример nginx-конфига

Ниже пример, если админам удобнее поднять сайт через nginx.

```nginx
server {
    listen 80;
    server_name <SUBDOMAIN>;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name <SUBDOMAIN>;

    ssl_certificate     /path/to/fullchain.pem;
    ssl_certificate_key /path/to/privkey.pem;

    root /var/www/<SUBDOMAIN>;
    index index.html;

    add_header X-Robots-Tag "noindex, nofollow, noarchive, nosnippet, noimageindex" always;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location = /robots.txt {
        try_files $uri =404;
    }
}
```
